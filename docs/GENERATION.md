# Generated API client

The Management API client (`pkg/apis/management/v1/`) is generated from the
upstream Lakekeeper OpenAPI spec. Everything else in the repo â€” including the
`pkg/client` facade, the `lkctl` CLI, and the storage / permissions builder
packages â€” is hand-written.

This document is the single source of truth for **what is generated**, **how
to regenerate it**, and **why a YAML preprocessor is involved**.

## What is generated

Everything under [`pkg/apis/management/v1/`](../pkg/apis/management/v1/) is
regenerated wholesale on every `make generate` run. The Makefile `rm -rf`s
the directory before invoking the generator (see
[Makefile:159](../Makefile#L159)), so any hand-edits there are silently
discarded the next time someone regenerates.

Files carrying this header are generator output and **must not be
hand-edited**:

```go
// Code generated by OpenAPI Generator (https://openapi-generator.tech);
// DO NOT EDIT.
```

The package's own [`.openapi-generator-ignore`](../pkg/apis/management/v1/.openapi-generator-ignore)
only suppresses files we don't want emitted into the repo at all
(`README.md`, `git_push.sh`, `.travis.yml`, `api/openapi.yaml`). It does
**not** preserve hand-edited files â€” there are none.

The accompanying [`.openapi-generator/FILES`](../pkg/apis/management/v1/.openapi-generator/FILES)
manifest enumerates every file the generator owns, so a regeneration that
adds or removes files produces an obvious diff.

## How to regenerate

```sh
make generate
```

The pipeline (see [Makefile:151-174](../Makefile#L151-L174)):

1. `go run ./api/openapi/preprocess` rewrites
   `api/openapi/management-open-api.yaml` â†’
   `api/openapi/management-open-api.processed.yaml`. See
   [The preprocessor](#the-preprocessor) below.
2. `rm -rf pkg/apis/management/v1` and recreate the directory.
3. Copy the top-level [`.openapi-generator-ignore`](../api/openapi/.openapi-generator-ignore)
   into the output directory.
4. Run `openapitools/openapi-generator-cli:v7.10.0 generate` in a container.
   Input spec (`-i`): the preprocessed
   `api/openapi/management-open-api.processed.yaml` from step 1.
   Generator config (`-c`):
   [`api/openapi/management-config.yaml`](../api/openapi/management-config.yaml).
5. Delete the processed YAML.
6. `goimports -w` + `go mod tidy` + `make fmt`.

The container engine resolves to `docker` or `podman` depending on what's on
PATH (see [Makefile:69](../Makefile#L69)), so `make generate` works on either.

The generator config is intentionally minimal â€”
[`management-config.yaml`](../api/openapi/management-config.yaml) lists only
the options that deviate from upstream defaults, with an inline comment
explaining each.

## Updating the spec

The committed [`management-open-api.yaml`](../api/openapi/management-open-api.yaml)
is the source of truth. Update it by replacing the file with a fresh export
from the Lakekeeper server (`/api-docs/openapi.json`, converted to YAML),
then run `make generate` and commit the spec and the regenerated tree in the
**same commit** so the diff stays reviewable.

## The preprocessor

Located at [`api/openapi/preprocess/`](../api/openapi/preprocess/). The
package doc on [`main.go`](../api/openapi/preprocess/main.go#L1-L37) is the
authoritative description of the transformation phases â€” read it before
extending the preprocessor.

The short version: openapi-generator cannot represent
`allOf(oneOf, extra-properties)`. It collapses such constructs into a single
struct holding the union of all variant fields, breaking both marshal
(zero-valued fields ship on the wire) and unmarshal (correct payloads get
rejected for missing fields they shouldn't contain). The preprocessor walks
`components.schemas` to flatten these constructs into named leaf schemas
with disambiguating discriminators, so the generator can emit clean union
types.

This is what unlocks the correct wire format for `StorageCredential` (commit
`cb74e97`) and the `*Assignment` families (commit `372cdc9`).

The preprocessor also runs two ergonomic passes that the generator alone
cannot deliver:

- **Phase 5 â€” strip `'null'` from `type: [<X>, 'null']` arrays.** OAS 3.1
  nullable type-arrays trigger `NullableX` 200-line wrappers in the Go
  generator output. Stripping the `'null'` member produces plain `*X`
  fields with `omitempty` instead. The wire format on absent values is
  identical; the lost capability is sending an explicit JSON `null`
  distinct from absent. Lakekeeper has no fields where that distinction is
  load-bearing today; for the ones that ever do, `keepNullable` in the
  preprocessor is a per-field opt-out list.
- **Phase 6 â€” inject `x-enum-varnames` on every named enum schema.** The
  Go generator's default constant naming
  (`<UPPER_SCHEMA>_<UPPER_VALUE>` â€” e.g. `S3FLAVOR_AWS`) is not Go
  idiomatic. The preprocessor synthesises
  `<PascalSchema><PascalValue>` per value (`S3FlavorAws`) and the
  generator config drops `enumClassPrefix`. The synthesised varnames
  preserve cross-enum uniqueness, so values that repeat across enums
  (`create`, `delete`, `select` on several `*Action` enums) still produce
  distinct constants like `ProjectActionCreate` and `ServerActionCreate`.

When the spec adds a **new** `allOf(oneOf, extras)` shape and the round-trip
tests in [`pkg/apis/management/v1tests/`](../pkg/apis/management/v1tests/)
start failing, the preprocessor is the place to extend.

## Generator configuration

[`management-config.yaml`](../api/openapi/management-config.yaml) keeps a
small set of knobs flipped from openapi-generator's defaults. The current
choices, with the intent behind each:

| Key | Value | Why |
|---|---|---|
| `useOneOfDiscriminatorLookup` | `true` | Lets the generator emit unambiguous unmarshal logic when the preprocessor adds a `discriminator` block |
| `enumClassPrefix` | `false` | Pairs with the preprocessor's `x-enum-varnames` injection; without this, the prefix would stack and produce names like `S3FLAVOR_S3FlavorAws` |
| `generateMarshalJSON` | `false` | Default Go encoding via the struct's `json:` tags is byte-identical to the custom `ToMap`-driven `MarshalJSON` (every optional field carries `omitempty`). Saves ~8 lines per model file with no wire-format change |
| `generateUnmarshalJSON` | `true` | Kept on so inbound responses still error on a missing required field. The price is rejecting payloads with unknown fields (`DisallowUnknownFields`); we accept that because Lakekeeper does not introduce unknown fields without a spec bump that the SDK regenerates against |
| `generateInterfaces` | `false` | We expose the generated client through `pkg/client`, so the additional `XxxAPI` interface variants are not needed |
| `withGoMod`, `isGoSubmodule` | `false`, `true` | Embed the generated package inside this module instead of producing a standalone Go SDK |
